EIP-1559

×

Status

Live

Summary

EIP-1559 implemented a hybrid transaction fee mechanism in Ethereum, instituting a base fee per block that was burned and a priority fee for validator incentivization. The base fee was dynamically adjusted each block, targeting a consistent 50% capacity utilization to reduce transaction fee volatility and improve network efficiency. Designed to create a more predictable fee structure, this approach aimed to align Ethereum's economic activity with its token value by diminishing supply, while ensuring validators were compensated for including transactions through the priority fee.

There are aspirations to enhance EIP-1559 in the future.

Account Abstraction

×

Status

Implementation is underway, and enshrinement research is ongoing

Summary

ERC-4337 has emerged as a standardized solution in Ethereum for account abstraction, utilizing an entry point contract to avoid direct modifications to the consensus layer. It enables users to encapsulate their intended actions in UserOperation objects, subsequently aggregated into bundles by miners or bundlers for inclusion in the blockchain. Notably, ERC-4337 introduces a mechanism for gas fee payments using ERC-20 tokens, such as USDC, via a paymaster system. This approach is not merely theoretical; it's actively implemented in projects like Eth-Infinitsm and Stackup.

Should ERC-4337 gain wider acceptance and demonstrate its efficacy, some form of embedding it into the protocol via an EIP will be pursued. This advancement would require broad consensus within the Ethereum community and thorough testing.

Ethereum Object Format (EOF)

×

Status

The (8) EIPs are written, and were sought for inclusion in Shanghai. Look for EOF to ship in an execution layer hard fork after Cancun.

Summary

Ethereum Object Format (EOF) is inspired by the concept of "object file formats" in traditional programming, where an object file is a structured representation of machine code and data that's ready for use by the system. EOF aims to bring a similar structured approach to the EVM, organizing the bytecode of smart contracts into distinct, systematic sections for code, data, and other types. This structure contrasts with the current format, where such distinctions are less clear and systematic.

In EOF, this structured format would not only allow for more modular and efficient processing of smart contracts within the EVM, but would also introduce versioning like object file formats in computing. The format would support new functionalities like improved control flow operations and stack validation, and enhance the overall efficiency and robustness of Ethereum's infrastructure.

Privacy, VDFs, and More

×

Status

Ongoing research

Summary

VDFs are computations that take a predetermined time to complete and cannot be sped up through parallel processing. This feature could ensure more secure and unpredictable randomness, crucial for validator selection. Deep cryptographic methods are also under consideration for integration into Ethereum's framework. This includes exploring obfuscation for enhanced privacy, fully homomorphic encryption for secure and private computation, and one-shot signatures to eliminate the need for slashing mechanisms in PoS.

Additionally, the exploration of delay-encrypted mempools targets the improvement of transaction privacy and the reduction of MEV and censorship risks. This involves encrypting transactions in the mempool for a specified period.

Also included here are practical solutions for "dust accounts," which contain funds too small to move due to gas fees outweighing their value .

It too provided a massive user experience upgrade by streamlining gas estimations. Various privacy focused roadmap items are also bunched in here, further strengthening the argument that The Splurge is The UX Tract.
